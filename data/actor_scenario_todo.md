# Actor / 场景推进 — 待办提炼

> 从「本地 vs API」「复读」「对话不推进」「gal 式变线」等讨论整理。用于后续实现与排期。

## 1. 问题与痛点

- **复读**：actor 多轮输出字面或语义上高度相似的 user 侧内容（体感「六遍同一句/同一意图」），对话像在空转。
- **推进感缺失**：期望「连续对话有进度」，但当前更像围绕同一命题车轱辘，缺少可验收的「每轮新增」。
- **意图澄清**：大模型在架构里应负责**导演**还是**台词与变奏**，需要与产品形态对齐（已倾向：图/状态管推进，LLM 管在合法步内说人话）。

## 2. 目标愿景（可对照验收）

- **场景驱动**：人定义可测场景（可先用结构化 case，长期可探索「自然语言 → spec」或辅助生成）。
- **演绎而非空转**：每轮有明确「合法下一步」（对目标施压维度、子目标、或对上一轮 assistant 的反馈），**禁止**无限同义循环。
- **gal 式/条件变线**（若采用）：**target 的回复**或 **judge 信号** 驱动节点转移，不同节点下**可说的话/可做的动作**不同，而不是只跑满 `max_turns`。

## 3. 与当前实现的差距（简要）

- 场景以 **YAML 为主**（`goal` / `states` / `allowed_actions`），不是「口播一整段即场景」；若要坚持口述场景，需另设 **落 spec 的桥梁**。
- 去重与防复读：**字面相近**与**多轮前历史**、**语义相似** 覆盖仍不完整；runner 有「与上一轮完全相同」的兜底，actor 内仅有短窗口 + 一次重试。
- **状态/剧情**：`states` 在 spec 中多为描述；**严格状态转移**（且由 assistant 或 judge 触发）**尚未**作为硬逻辑跑通（与「target 说话转变内容」仍有距离）。

## 4. 待实现方向（可拆需求）

### 4.1 推进与防轱辘（高优先）

- 将「推进」**外显**为可执行规则：每轮必须满足的条件（如：必须包含对上一轮 assistant 的引用/概括；或必须切换子目标/动作类）。
- 扩大 **反重复** 范围：全对话 user 句列表、或更粗的语义/hash 去重，避免仅靠「最近 3 轮 + 与上一轮比」。
- 可选：**禁止连续两轮同一 `chosen_action`** 或与 case 表联动禁止「回退」到已完成的子目标（按产品设计取舍）。

### 4.2 状态机 / 有向图（中优先，对应 gal 感）

- 用 **节点 + 带条件的边** 表达章与分支；边上条件来自 target 文本轻量规则、**judge 标签**、或专门分类器。
- 每节点维护 **允许的用户意图/台词条目**；LLM 仅在当前节点**合法集合**内生成或润色，减少导演职责失控。

### 4.3 LLM 角色边界（中优先，文档/接口）

- 在 README 或设计小节写清：**图/条件拥有「下一步去哪」**；**actor LLM 拥有「在合法一步内怎么说」**（及有限开放域的变奏/施压措辞）。
- 评估 **纯模板 actor** 与 **LLM 受限生成** 的适用边界（可复现 / 成本 / 自然度）。

### 4.4 与「口播场景」对齐（低优先/探索）

- 若需要「我讲一段，系统就能跑」：定义最小 **场景 DSL** 或半自动从自然语言生成 `goal` / 状态 / 边 / 池子，与现有 `cases/*.yaml` 对接。
